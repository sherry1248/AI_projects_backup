# STS2 插件框架设计案

## 1. 产品定位

`sts2_autoplay` 的主定位不再是“重型自动游玩插件”，而是 **陪玩优先、教练优先** 的杀戮尖塔 2 游戏协作插件。

它的主要价值应当是：

- 持续理解当前局势
- 用适合 N.E.K.O 的方式进行点评、提醒和建议
- 在关键抉择节点给出高价值陪玩反馈
- 记录并应用用户的结构化偏好
- 在用户明确授权或场景足够低风险时，提供有限、稳定、可解释的自动执行

这意味着：

- **陪玩是主线能力**
- **自动游玩是辅线能力**
- 自动游玩保留，但不再作为产品核心方向持续加重复杂度
- 不再把“更像人、更强自主 AI”当作主要演进目标

新的核心定义是：

> STS2 是一个会看局、会讲局、会提醒、会建议、必要时能帮你点两下的陪玩插件，而不是一个以重型 AI 自主通关为中心的自动体。

---

## 2. 体验模型

STS2 的核心体验应围绕如下链路展开：

1. **Observe**：读取游戏状态、识别当前场景与合法动作
2. **Understand**：构建局势上下文、策略上下文、偏好上下文
3. **Summarize**：生成结构化局势摘要与重点变化摘要
4. **Comment**：按节奏输出陪玩点评，而不是机械播报所有状态
5. **Remind**：在风险点、遗忘点、关键节点给出提醒
6. **Suggest**：在高价值决策点给出推荐动作与理由
7. **Optionally Act**：在低风险、重复性、用户授权明确时执行启发式动作

这里的重点不是“系统替玩家思考一切”，而是：

- 在多数时间里，它是一个**懂局势的陪玩者**
- 在少数适合的时候，它才是一个**保守的辅助执行器**

---

## 3. 核心设计原则

### 3.1 陪玩优先

所有架构决策默认优先服务于：

- 更好的局势理解
- 更好的陪玩表达
- 更好的提醒和建议
- 更低打扰、更高相关性的猫娘协同

如果某项设计明显提升自动游玩复杂度，却不能同步提升陪玩体验，就不应继续作为主框架目标。

### 3.2 策略文档优先于重型决策模型

策略文档是 STS2 的核心资产。

它不应只被视为“喂给模型的 prompt 材料”，而应成为整个插件共享的策略知识底座：

- 陪玩点评读取它
- 提醒逻辑读取它
- 推荐逻辑读取它
- 自动执行也读取它

也就是说，策略文档要从“模型附件”升级为“系统规则知识源”。

### 3.3 自动游玩启发式优先

自动游玩默认应当是：

- 策略文档驱动
- 启发式优先
- 保守执行
- 可解释
- 可停下

而不是：

- 多模式复杂切换
- 重型 AI 主导
- 高维护成本的自主决策体系

### 3.4 所有动作必须统一收口

无论动作来自：

- 启发式规划
- 用户授权执行
- 陪玩建议转执行
- 未来保留的模型增强层

最终都必须经过统一动作执行边界：

- 合法性匹配
- 参数补全
- 执行前重校验
- 真正 dispatch 到 STS2-Agent

### 3.5 selective sync 而不是全量播报

不是所有状态变化都值得让猫娘知道，更不是所有内部判断都应该变成用户可见发言。

只同步：

- 高价值局势变化
- 值得点评的节点
- 值得提醒的风险
- 值得建议的关键选择

低价值轮询、重复变化、内部执行细节应尽量留在插件内部。

### 3.6 用户偏好必须结构化

用户对事件、选牌、删牌、路线、战斗风格的补充偏好，不应只存在于自然语言上下文里。

它们应当：

- 分域存储
- key 覆盖写入
- 能被推荐层和自动执行层共同读取

### 3.7 standby 是陪玩待机，不是系统消失

standby 不等于 shutdown。

standby 下：

- 停止主动执行动作
- 保留状态刷新
- 保留局势整理
- 保留摘要构建
- 保留猫娘读取与建议能力

也就是说，待机时仍然应该是一个“能看局、能说局、但不乱动手”的陪玩系统。

## 4. 新架构总览

新的 STS2 框架应当分成四条主链：

### 4.1 状态理解链

`transport_client.py`
→ `snapshot_normalizer.py`
→ `state_machine.py`
→ `summary_context_builder.py`

职责：

- 拉取原始游戏状态
- 归一化状态与合法动作
- 给出当前状态语义
- 提取适合陪玩和决策的结构化上下文

### 4.2 策略与偏好链

`strategies/*`
→ `strategy_parser.py`
→ `strategy_repository.py`
→ `preference_store.py`
→ `preference_extractors.py`

职责：

- 提供角色/场景策略知识
- 将策略按状态选择性装配
- 将用户偏好结构化保存
- 将偏好融合进推荐与执行逻辑

### 4.3 陪玩输出链

`summary_context_builder.py`
→ `situation_summary_engine.py`
→ `catgirl_bridge.py`
→ `neko_interface.py`

职责：

- 从局势中提取重点
- 生成结构化摘要与变化摘要
- 决定是否值得同步给猫娘
- 支持猫娘查询、指导注入和外部协同

### 4.4 有界自动执行链

`strategy_repository.py`
→ `heuristic_planner.py`
→ `candidate_generator.py`（可弱化）
→ `action_engine.py`
→ `service.py / loop_runner.py`

职责：

- 在已知状态下根据策略和偏好生成保守动作
- 仅在适合时自动执行
- 将执行边界限制在可解释、可重试、可停下的范围内

## 5. 模块职责定义

### 5.1 `service.py`

**定位：顶层编排中枢**

负责：

- lifecycle
- 状态刷新
- 自动执行状态管理
- standby 与 companion mode 管理
- 同步、摘要、执行链的统一编排

它不应该承载大量具体策略细节，而应作为总调度中心。

### 5.2 `state_machine.py`

**定位：唯一状态真相源**

负责：

- 识别当前 screen / state_name / screen_class
- 输出执行与摘要所需状态元信息
- 为后续推荐、陪玩、执行链提供统一状态语义

后续不应把状态判断再次散落回 service 或 planner 内部。

### 5.3 `summary_context_builder.py`

**定位：结构化局势提取器**

负责：

- 为不同状态提取不同数据块
- 生成摘要输入
- 生成推荐输入
- 生成提醒输入
- 生成策略应用上下文

它是陪玩质量的关键模块之一。

### 5.4 `situation_summary_engine.py`

**定位：局势摘要引擎**

负责：

- 生成当前局势摘要
- 生成前后变化摘要
- 保持摘要结构清晰、信息密度高

它的目标不是“自己扮演猫娘”，而是为猫娘表达提供可靠摘要原料。

### 5.5 `strategy_repository.py`

**定位：策略知识装配中心**

负责：

- 根据角色和状态选取策略文档片段
- 加载 constraints / directives
- 叠加结构化偏好
- 产出给推荐层和自动执行层可消费的策略上下文

以后这里应该越来越像“规则知识仓库”，而不是“给模型凑 prompt 的工具”。

### 5.6 `preference_store.py` / `preference_extractors.py`

**定位：结构化偏好层**

负责：

- 存储偏好
- 覆盖更新
- 按状态/域提取偏好
- 支持从猫娘指导或用户自然语言中抽取结构化偏好

### 5.7 `catgirl_bridge.py`

**定位：陪玩消息桥接层**

负责：

- selective sync
- 控制哪些内容只进 AI context
- 控制哪些内容值得猫娘主动表达
- 避免把所有游戏侧变化都推成可见消息

它应该严格区分：

- 被动上下文同步
- 主动点评触发
- 完全内部消息

### 5.8 `neko_interface.py`

**定位：猫娘/上层交互接口**

负责：

- 查询状态
- 查询摘要
- 查询策略上下文
- 注入指导
- 写入偏好
- 切换模式

它应保持“薄接口”风格，不承载重逻辑。

### 5.9 `heuristic_planner.py`

**定位：启发式推荐与执行规划器**

在新架构下，它应成为自动游玩默认主脑。

负责：

- 根据状态 + 策略 + 偏好生成推荐动作
- 给出简单可解释的执行理由
- 在低风险或单义场景下生成自动执行动作

它不再只是“模型前的 fallback”，而是应被提升为默认执行器。

### 5.10 `action_engine.py`

**定位：唯一动作执行边界**

负责：

- 匹配合法动作
- 规范 kwargs
- 保证动作可执行
- 统一 dispatch 到 STS2-Agent

任何自动执行能力都必须经过这里。

### 5.11 `loop_runner.py`

**定位：后台观察 + 有界执行支撑层**

职责应弱化为：

- 定期刷新状态
- 驱动必要的自动执行 tick
- 不再承担产品叙事核心地位

它是基础设施，不应继续被视为插件的主卖点。

### 5.12 `game_llm_agent.py` / `llm_decision_engine.py`

**定位：降级为可选增强层**

这两个模块可以保留，但不应再被视为主路径核心：

- 可用于难例解释
- 可用于未来实验性增强
- 可用于少量高歧义决策的辅助建议

但不再应被框架文档描述为 STS2 的中心执行引擎。

## 6. 自动游玩重定义

## 6.1 自动游玩的目标

自动游玩保留，但目标收缩为：

- 辅助执行重复性动作
- 辅助处理明显低风险动作
- 在用户明确开启后提供有限自动化
- 不以“完整自主通关”为主框架目标

## 6.2 自动游玩的主路径

新的推荐默认路径应为：

`state`
→ `strategy context`
→ `heuristic planner`
→ `action validation`
→ `optional execution`

这里的重点是：

- **策略文档驱动**
- **启发式优先**
- **执行边界稳定**
- **尽量可解释**

## 6.3 自动游玩的适用范围

优先适用于：

- 明显菜单动作
- 单义确认动作
- 低风险连续推进动作
- 有清晰策略偏好的常规选择

谨慎适用于：

- 高价值奖励抉择
- 事件分支选择
- 复杂路线决策
- 高风险战斗回合

这些高价值节点更应优先触发：

- 点评
- 提醒
- 建议
- 等用户决定

而不是无脑自动点。

## 6.4 模型增强的地位

模型增强可以存在，但只能是：

- 可选的
- 次级的
- 可关闭的
- 不影响主路径稳定性的

框架层面不再围绕：

- full-program
- half-program
- full-model
- 多重回退阶梯

继续扩展复杂度。

如果未来保留 mode，也应只是内部兼容概念，而不是产品主叙事。

---

## 7. 消息与猫娘协同模型

### 7.1 push_message 的三类用途必须严格区分

STS2 后续应明确区分三类消息：

1. **上下文同步消息**
   - 只进 AI context
   - 不一定对用户可见
   - 典型：`visibility=[]`, `ai_behavior="read"`

2. **陪玩表达触发消息**
   - 值得猫娘主动说一句
   - 典型：`visibility=[]` 或 chat 可见，`ai_behavior="respond"`

3. **内部静默消息**
   - 只用于插件内部状态记录或前端非人格化 UI
   - 典型：`ai_behavior="blind"`

### 7.2 陪玩点评的原则

陪玩点评不应是“每次轮询都说一句”，而应围绕：

- 关键变化
- 高风险局面
- 值得表扬/吐槽的节点
- 长时间未输出后的节奏补点

### 7.3 selective sync 原则

值得同步的内容：

- 局势发生明显变化
- 状态切换具有叙事价值
- 当前节点适合建议或提醒
- 风险明显上升

不值得同步的内容：

- 每次无变化轮询
- 所有内部中间决策
- 所有候选动作列表
- 所有执行细节日志

### 7.4 standby 语义

standby 下：

- 不自动执行动作
- 继续刷新状态
- 继续构建摘要
- 继续允许猫娘查询和建议
- 继续允许偏好写入与指导注入

这是“陪玩继续在线，自动手脚收起来”的模式。

---

## 8. 推荐的对外能力层次

### 第一层：陪玩基础能力（主推）

- 获取状态
- 获取快照
- 获取局势摘要
- 获取猫娘同步包
- 获取策略上下文
- 风险提醒
- 节奏点评
- 关键节点建议

### 第二层：偏好与指导能力（主推）

- 注入指导
- 更新偏好
- 查询偏好
- 列出偏好
- 让系统越来越懂用户打法

### 第三层：自动执行能力（次推）

- 获取规划动作
- 执行规划动作
- 开始/暂停/恢复/停止自动运行
- 待机切换

这一层应该保留，但不再是文档中的主叙事中心。

---

## 9. 迁移说明

### 9.1 不需要立刻删除的内容

以下内容可以保留，但要降级定位：

- `game_llm_agent.py`
- `llm_decision_engine.py`
- 复杂 mode 语义
- 候选动作 + 模型裁决路径

它们可以继续作为：

- 实验性增强
- 未来可选扩展
- 难例辅助路径

而不是框架主轴。

### 9.2 需要提升定位的内容

后续框架应更强调：

- `strategy_repository.py`
- `summary_context_builder.py`
- `situation_summary_engine.py`
- `catgirl_bridge.py`
- `heuristic_planner.py`
- `action_engine.py`

因为这些模块真正对应新的产品价值：

- 会看局
- 会讲局
- 会提醒
- 会建议
- 必要时稳定帮忙执行

### 9.3 文档叙事的变化

旧文档叙事偏向：

- 多模式自治
- 状态级 AI/程序路由
- 模型决策占据较高地位

新文档叙事应改成：

- 陪玩优先
- 策略知识优先
- 启发式自动执行优先
- selective sync 优先
- 模型增强可选而非中心

---

## 10. 最终框架结论

新的 STS2 框架不再追求把插件继续推向“更重的自动游玩系统”，而是要把它收敛成一个：

- **懂局势的陪玩插件**
- **依赖策略文档的建议系统**
- **带有限自动执行能力的保守教练系统**

它的核心能力不应再被定义成：

- 多模式 AI 自主打牌
- 越来越复杂的自动游玩路线

而应被定义成：

- 让猫娘更懂这一局
- 让用户更容易做对关键决策
- 在适合时机稳定地帮用户点掉低风险动作

这是更符合产品主推方向、也更适合长期维护的 STS2 框架设计案。
